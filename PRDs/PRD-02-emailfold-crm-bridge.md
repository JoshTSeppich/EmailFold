# PRD-02 — EmailFold → EventFold: Email Draft Persistence
**Version:** 1.0
**Date:** 2026-03-10
**Author:** Foxworks / Claude Architecture Session
**For:** Senior Dev (EventFold CRM) + EmailFold owner
**Priority:** P0 — Core daily workflow

---

## Problem

EmailFold generates cold emails for 10–20 companies per session. Every draft lives only in Electron localStorage. The user has to:
1. Manually copy the subject + body from EmailFold
2. Open Gmail/Outlook
3. Paste and send
4. Navigate to EventFold CRM and manually create an Interaction record
5. Set a reminder to follow up

Steps 3–5 generate zero CRM value and take 3–5 minutes per company. At 15 companies/session, that's 45–75 minutes of pure admin per prospecting session.

**The gap:** Emails generated by AI are invisible to the CRM.

---

## Proposed Solution

1. **EmailFold** adds a "Send to EventFold CRM" button on each email card. One click serializes the selected draft to the system clipboard as a typed JSON payload (`__emailfold_v1`).

2. **EventFold CRM** adds a "Paste from EmailFold" button in the Company Detail view and a new Outbox panel. One click reads the clipboard, shows a confirmation modal, and on confirm:
   - Finds or creates the **Company** record
   - Finds or creates the **Contact** record
   - Creates an **Interaction** record (type=Email, direction=Outbound, status=Draft) with the email body + subject + angle metadata
   - Creates a companion **Note** with the full AI research snapshot (all generated variants, signals, tech stack, pain points)

3. Once in the CRM, the email draft has a lifecycle: **Draft → Sent → Replied**. "Mark Sent" auto-creates a follow-up Task.

---

## User Stories

- As a SDR, I want to send an email draft to EventFold with one click so I don't manually log outbound emails
- As a SDR, I want to see all my email drafts in one Outbox view so I know exactly what to send today
- As a SDR, I want to click "Mark Sent" after sending an email so that a follow-up reminder is auto-created
- As a SDR, I want to log a reply against an email so the thread is tracked in the CRM
- As a manager, I want to see which EmailFold angles are driving replies so I can tune the prompts

## Clipboard Payload Contract

EmailFold writes this on "→ Send to CRM" click:

```typescript
interface EmailFoldBridgePayload {
  __emailfold_v1: true;
  companyName: string;
  companyUrl: string;
  contactName: string;
  contactRole: string;
  emailGoal: string;
  ts: number;                          // Unix ms — time of generation
  chosenAngleIndex: number;            // 0-based index into emails[]
  research: {
    company_name: string;
    industry: string;
    what_they_do: string;
    tech_stack: string[];
    recent_news: string;
    pain_points: string[];
    icp_fit: string;                   // "Strong" | "Moderate" | "Weak"
    personalization_hooks: string[];
    signals_confirmed: string[];
    signals_missing: string[];
  };
  emails: Array<{
    angle: string;
    subject: string;
    subjects: string[];                // 3 subject line variants
    body: string;
    hook_used: string;
    why: string;
  }>;
}
```

## Interaction Aggregate Extensions

Add these optional fields to `InteractionEvent::Created`. All `Option<T>` — backward-compatible with existing JSONL streams (serde skips missing on old events).

```rust
// In src/domain/interaction.rs — extend Created variant:
email_subject: Option<String>,
email_status: Option<EmailStatus>,
ai_generated: Option<bool>,
angle_name: Option<String>,

pub enum EmailStatus {
    Draft,
    Sent,
    Replied,
    Bounced,
}

// New event variants:
pub enum InteractionEvent {
    // ... existing variants unchanged ...

    /// Email was marked as sent to the recipient.
    EmailSent {
        sent_at: String,                     // ISO 8601
    },
    /// A reply was logged against this email thread.
    ReplyLogged {
        reply_summary: String,
        replied_at: String,
        reply_interaction_id: String,        // ID of new inbound Interaction
    },
    /// Email lifecycle status updated (e.g. Bounced).
    EmailStatusUpdated {
        status: EmailStatus,
    },
}
```

**Update `InteractionWithId` struct** in `commands.rs` to include these new optional fields so the frontend can render them.

## New Tauri IPC Commands

```rust
/// Import EmailFold draft from clipboard.
/// Orchestrates: find/create Company → find/create Contact →
///   create Interaction (Draft) → create companion Note → optionally create Task.
#[tauri::command]
pub async fn import_emailfold_draft(
    company_name: String,
    company_url: Option<String>,
    contact_name: String,
    contact_role: Option<String>,
    email_goal: String,
    email_subject: String,
    email_body: String,
    angle_name: String,
    occurred_at: String,               // ISO 8601 from payload.ts
    research_note_body: String,        // Pre-formatted markdown from frontend
    create_followup_task: bool,
    state: State<'_, AppState>,
) -> Result<ImportEmailFoldResult, AppError>

pub struct ImportEmailFoldResult {
    pub interaction_id: String,
    pub contact_id: String,
    pub company_id: String,
    pub note_id: String,
    pub task_id: Option<String>,
    pub company_created: bool,
    pub contact_created: bool,
}

/// Mark an email interaction as sent. Transitions Draft → Sent.
/// Creates a follow-up Task if contact_id / company_id available.
#[tauri::command]
pub async fn mark_email_sent(
    id: String,                       // Interaction ID
    sent_at: String,                  // ISO 8601, defaults to now
    state: State<'_, AppState>,
) -> Result<(), AppError>

/// Log an inbound reply to an existing email thread.
/// Creates a new Inbound Email Interaction, links to original.
#[tauri::command]
pub async fn log_email_reply(
    original_interaction_id: String,
    reply_summary: String,
    reply_body: Option<String>,
    replied_at: String,
    state: State<'_, AppState>,
) -> Result<String, AppError>          // returns new Interaction ID

/// Find a contact by name within a company.
#[tauri::command]
pub async fn find_contact_by_name_in_company(
    name: String,
    company_id: String,
    state: State<'_, AppState>,
) -> Result<Option<String>, AppError>  // returns contact_id

/// List email interactions for a company (for Outbox).
#[tauri::command]
pub async fn list_email_drafts_for_company(
    company_id: String,
    state: State<'_, AppState>,
) -> Result<Vec<InteractionWithId>, AppError>
```

## Out of Scope

- Actually sending the email from EventFold (user still sends via Gmail/Outlook)
- Email open/click tracking (requires hosted infrastructure)
- Two-way sync (EventFold → EmailFold)
- Auto-detecting when EmailFold copies to clipboard (user clicks the button manually)

## Success Metrics

- Time from email generation → logged in CRM: **< 60 seconds**
- Follow-up task creation rate: **100%** (auto, zero extra steps)
- Manual CRM logging drops to zero for EmailFold-generated outreach

## Open Questions for Senior Dev

1. Does `Interaction.body` have a length limit in the JSONL store? A full email body is ~300 words. The companion Note body (research snapshot) is ~800 words.
2. The `mark_email_sent` command needs to read the Interaction's current state to get `contact_id`/`company_id` for Task creation. Does `AppState` expose a `read_aggregate_state(type, id)` method, or does the command need to replay events?
3. Should the Draft/Sent/Replied status filter in Company Detail be added to the existing interactions query, or should `list_email_drafts_for_company` be a separate endpoint that's only used by the new Outbox?
